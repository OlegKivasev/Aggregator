# План первой версии агрегатора автозапчастей

## Цель

Сделать внутреннее web-приложение, которое по введенному артикулу ищет товары у нескольких поставщиков, показывает результаты по мере поступления и позволяет быстро перейти к найденной позиции на сайте поставщика.

## Что входит в первую версию

- Внутреннее `web-приложение`
- Стек: `Node.js + TypeScript + Playwright`
- Поиск только по `одному артикулу`
- Поиск у нескольких поставщиков `параллельно`
- Показ результатов `по мере поступления`
- Использование `SSE` для streaming-ответа
- Сортировка результатов только `по цене`
- Быстрый переход на найденную позицию у поставщика
- Работа с `корпоративными аккаунтами` поставщиков
- `Постоянные авторизованные сессии` по каждому поставщику
- `Таймаут 15 секунд` на одного поставщика
- Если один поставщик не ответил, остальные результаты все равно показываются

## Что не входит в первую версию

- База данных
- Кэш
- История поисков
- Авторизация пользователей приложения
- Поиск по VIN
- Поиск по названию
- Аналоги и кроссы
- Группировка результатов
- Дедупликация
- Корзина
- Оформление заказа
- Массовый поиск по списку артикулов

## Зафиксированные продуктовые решения

### Сценарий пользователя

1. Пользователь вводит артикул.
2. Система запускает поиск у всех поставщиков параллельно.
3. Результаты показываются частями, по мере готовности.
4. Пользователь видит, что поиск еще продолжается.
5. Пользователь получает плоский список результатов.
6. Пользователь может нажать на результат и быстро перейти к нужной позиции на сайте поставщика.

### Логика поиска

- Пользователь вводит только `артикул`
- Бренд пока `необязателен` и в интерфейсе не требуется
- Показываются только `точные совпадения по артикулу`
- Аналоги и кроссы в первой версии `не показываются`
- Группировка результатов `не делается`
- Дедупликация `не делается`
- Результаты выводятся `как вернул поставщик`

### Поля результата в интерфейсе

- `поставщик`
- `артикул`
- `наименование`
- `цена`
- `срок`
- `ссылка` или быстрый переход к найденной позиции

### Сортировка

- Только `по цене`

### Поведение при ошибках

- Ошибка одного поставщика `не должна` ломать весь поиск
- Для каждого поставщика должен отображаться отдельный статус
- Возможные статусы:
  - `в поиске`
  - `завершен`
  - `таймаут`
  - `ошибка авторизации`
  - `ошибка`

## Поставщики

Планируемые поставщики:

- `Armtek`
- `Rossko`
- `Part-Kom`
- `STParts`

Порядок подключения:

1. `Rossko`
2. Второй поставщик
3. Остальные поставщики

## Технический подход

### Общий принцип интеграции

Для каждого поставщика:

1. Сначала проверяем, можно ли использовать `открытый API` или `внутренние XHR/API-запросы`
2. Если это невозможно или нестабильно, используем `Playwright`

### Архитектурные блоки

- `frontend` с формой поиска и списком результатов
- `backend` с endpoint для streaming-поиска
- `SSE` для передачи результатов по мере поступления
- `supplier adapters` для каждого поставщика
- `session manager` для постоянных авторизованных сессий
- `result normalizer` для приведения ответов к единому формату
- `timeout/error handling` для единообразной обработки ошибок

### Модель выполнения поиска

- Один запрос пользователя запускает один живой поиск
- Ничего не сохраняется между запросами
- Нет БД и нет постоянного хранения результатов
- Нет кэша
- Логика первой версии: `запрос -> поиск -> поток результатов -> завершение`

## Развертывание

- Все приложение работает на `одной внутренней машине/сервере`
- `backend`, `frontend` и `Playwright` запускаются централизованно
- Пользователи не запускают систему локально

## Операционные требования

- Нужен простой внутренний механизм `check session / relogin`
- Должна быть возможность вручную проверить, жива ли сессия поставщика
- Должна быть возможность вручную переподключить сессию поставщика

## Критерий готовности первого этапа

Первый этап считается завершенным, когда `Rossko` работает end-to-end:

1. Пользователь вводит артикул в web-интерфейсе
2. Backend запускает поиск по `Rossko`
3. Результаты приходят без ручного вмешательства
4. Frontend показывает их в реальном времени
5. Пользователь может нажать на строку и перейти к нужной позиции
6. При ошибке показывается понятный статус

## Этапы реализации

### Этап 1. Каркас приложения

- Создать `frontend`
- Создать `backend`
- Реализовать форму поиска
- Реализовать потоковую выдачу через `SSE`
- Определить единый формат результата поставщика

### Этап 2. Базовая архитектура интеграций

- Создать интерфейс `SupplierAdapter`
- Создать общий контракт результата
- Создать базовую обвязку для таймаутов и ошибок
- Создать менеджер сессий поставщиков

### Этап 3. Первый поставщик: Rossko

- Исследовать поиск по артикулу
- Проверить наличие внутреннего API/XHR
- Если API неудобен или отсутствует, реализовать сценарий через `Playwright`
- Поднять постоянную авторизованную сессию
- Получать результаты и стримить их в интерфейс
- Реализовать быстрый переход к найденной позиции

### Этап 4. Служебные инструменты

- Добавить `check session`
- Добавить `relogin`
- Добавить минимальную диагностику ошибок по поставщику

### Этап 5. Второй поставщик

- Подключить второго поставщика на том же архитектурном каркасе
- Проверить, что архитектура адаптеров действительно переиспользуется

### Этап 6. Остальные поставщики

- Подключить остальных поставщиков по очереди
- Унифицировать поведение поиска, статусов и переходов

## Основные риски

- Разлогин и протухание сессий
- Капча или антибот-защита
- Изменение интерфейса сайтов поставщиков
- Отсутствие удобной прямой ссылки на карточку товара
- Разные структуры выдачи у разных поставщиков
- Медленные ответы отдельных сайтов
- Хрупкость чистой UI-автоматизации

## Практический приоритет на старт

1. Поднять каркас `frontend + backend + SSE`
2. Определить интерфейс `SupplierAdapter`
3. Начать интеграцию с `Rossko`
4. Сначала искать внутренний API/XHR
5. Только потом переходить к полноценной browser automation через `Playwright`

## Короткий итог

Первая версия должна быть максимально простой и быстрой в реализации:

- без БД
- без кэша
- без сложной бизнес-логики
- без группировки
- без заказа

Главная цель первой версии: быстро получить рабочий внутренний инструмент, который реально ищет по артикулу у нескольких поставщиков и экономит время менеджеру.
